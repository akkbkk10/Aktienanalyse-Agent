# Fact Report Expectations

This document defines narrow expectations for generated fact report Markdown
artifacts. It documents current behavior and review boundaries only. It does
not add a Markdown parser, schema, validator, runtime behavior, test behavior,
or report wording change.

Generated fact reports remain user-facing Markdown artifacts. Machine-readable
contracts for embedded DCF, fair value per share, model rating, model
confidence, model signal, and analysis summary outputs remain in their
standalone JSON contracts.

## Artifact Role

The fact report is the per-ticker Markdown report generated by
`scripts/generate_report.py` and written under the configured reports
directory:

```text
<reports-dir>/<TICKER>/<TICKER>_fact_report.md
```

The fact report is intended for human review. It summarizes validated facts,
calculated ratios, source references, optional model/calculation outputs,
missing data, warnings, and output boundaries. It must remain deterministic,
evidence-based, and no-investment-advice.

Generated reports must remain under ignored `reports/` paths and must not be
committed.

## Stable Expectations

Current generated fact reports are expected to include these stable
report-level areas:

- report title
- run metadata
- facts
- validation status
- calculated ratios
- source references
- missing data
- warnings
- boundary / no-investment-advice framing

The report title is expected to identify the ticker:

```text
# <TICKER> Fact Report
```

Run metadata is expected near the top of the report and currently includes:

- generated timestamp
- ticker
- audit log reference

The facts area is expected to keep validation status, calculated ratios, and
source references separate from missing data, warnings, and boundary language.

## Optional Calculation And Model Sections

These sections may appear when the corresponding upstream artifact is available
to report generation:

- DCF calculation output
- fair value per share model calculation
- model rating
- model confidence
- model signal

When present, calculation and model sections should remain clearly labeled as
model or calculation output. They should not be rewritten into investment
advice, price targets, buy/sell/hold recommendations, broker/order guidance, or
portfolio guidance.

When DCF output is present, assumptions should remain clearly labeled and
separate from sourced facts. Current generated reports use an assumptions
subsection under the DCF calculation output.

## Flexible Content

The exact Markdown prose and detailed content should remain flexible. These
areas should not be frozen by this expectations document:

- exact wording and sentence structure
- source-reference details
- warning text
- model confidence reasons
- model signal reasons
- model signal blocking reasons
- timestamps
- paths and audit log references
- financial values
- ratio ordering
- DCF assumptions and scenario details
- fair value per share scenario details
- model rating labels and assumptions
- research-gap wording
- embedded upstream output details

Future wording changes should preserve the section intent, source traceability,
and no-investment-advice boundaries rather than matching exact current prose.

## Guardrail Expectations

Generated fact reports must not include:

- buy/sell/hold recommendations
- price targets
- personal investment advice
- broker/order instructions
- automated trading language
- portfolio automation guidance
- invented sources
- unsupported live-fetching claims

Generated fact reports should make these items visible when applicable:

- assumptions, clearly labeled
- missing data
- warnings
- source references
- audit log reference
- boundary/disclaimer language

The fact report may include existing no-investment-advice disclaimers from
embedded model/calculation outputs. Those disclaimers do not make the report an
advice artifact.

## Why There Is No Parser, Schema, Or Validator Yet

No Markdown parser, schema, or validator is added now because the fact report is
prose-heavy and user-facing. The stable boundary is the report role, path,
section intent, and guardrails, not every line of Markdown text.

Parser-backed validation would add complexity and could over-constrain safe
wording changes. It could also duplicate lower-level JSON contracts that already
protect machine-readable DCF, fair value per share, model rating, model
confidence, model signal, and analysis summary artifacts.

Current protection remains:

- report generation from structured inputs
- prohibited-language checks in `scripts/generate_report.py`
- report-focused tests in `tests/test_generate_report.py`
- forbidden-output regression tests
- generated-output review guidance
- report artifact path tests in the v1.0 demo suite

## When A Future Contract Could Be Justified

A future heading or section-level contract may become justified if:

- report sections drift repeatedly without intentional review
- downstream documentation or review tools depend on stable headings
- generated-output review finds missing boundary, missing data, or warning
  sections
- adapter work increases report variation enough that manual expectations are
  no longer sufficient
- maintainers need automated checks for section presence without validating
  exact prose

Even then, a future contract should start with heading or section presence only.
It should not validate exact wording, warning text, reasons, blocker text,
financial values, timestamps, source URLs, paths, or embedded upstream output
internals unless a separate assessment shows that stricter validation is needed.

## Out Of Scope

This expectations document does not add or change:

- Markdown parser implementation
- schema implementation
- validator implementation
- runtime code
- tests
- CI
- report wording
- financial logic
- model rating, model confidence, or model signal behavior
- DCF or fair value logic
- audit log schema behavior
- generated reports
- release notes
- tags
- live fetching
- adapters or framework code
- investment advice
- trading or portfolio logic
