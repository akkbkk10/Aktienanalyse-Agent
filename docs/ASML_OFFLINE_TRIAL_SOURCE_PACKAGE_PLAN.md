# ASML Offline Trial Source Package Plan

This document plans the first real offline company-data trial for ASML Holding
N.V. (`ASML`). It is a planning document only. It does not add ASML source data,
download source material, fetch live data, add adapters, call provider APIs,
change schemas, change config, change runtime behavior, change financial logic,
create generated reports, or publish a release.

The trial type is **manual offline source package only**. Any ASML figures,
source URLs, source dates, market price snapshots, share counts, assumptions,
or generated outputs must be added later through a separate reviewed PR.

Use `docs/ASML_SOURCE_MATERIAL_HANDOFF_CHECKLIST.md` before creating that PR.
It lists the official source materials and metadata that must be manually
provided before ASML data files can be prepared.

## Purpose

The ASML trial should prove that a manually prepared ASML source package can
move through the existing deterministic evidence, validation, analysis,
generated-artifact, audit-log, and guardrail workflow without introducing live
data, provider APIs, adapters, recommendations, price targets, trading logic, or
portfolio automation.

The trial should answer only these questions:

- Can ASML source records satisfy the existing financial metric contract?
- Can ASML use the existing onboarding and offline-trial commands?
- Are ASML-specific source, currency, accounting-basis, and market-snapshot
  choices clear enough for manual review?
- Do generated artifacts remain inspectable and guardrail-compliant when ASML
  data is provided later?

## Explicit Non-Implementation Boundary

This plan does not authorize:

- ASML data files
- ASML context files
- ASML watchlist entries
- ASML DCF assumptions
- source scraping or downloading
- live market data fetching
- provider API integration
- adapter implementation
- MCP/A2A runtime integration
- web UI or desktop app behavior
- generated reports committed to the repository
- financial logic or valuation formula changes
- model rating, confidence, or signal behavior changes
- report wording changes
- buy/sell/hold recommendations
- price targets
- personal investment advice
- trading logic
- portfolio automation

- IFRS accounting-basis labels

- manually captured market price snapshot metadata
- source-date and last-verified freshness

Use the generic templates as the starting shape:

- `data/company_template/sample_metrics_template.json`
- `data/company_template/dcf_assumptions_template.json`

Use existing sample packages only as shape examples, not as ASML data sources:

- `data/tsmc_sample_metrics.json`

## Required Local Input Files For A Future Trial

A later ASML source-package PR should prepare explicit local files such as:

- `data/asml_sample_metrics.json`
- `data/companies/ASML/dcf_assumptions.json`, only if DCF is included in the
  trial
- a proposed `ASML` entry in `config/watchlist.json`
- generated `data/companies/ASML/context.json` only after source validation
  passes and the onboarding workflow requires it

For a temporary local trial before an onboarding PR, generated contexts, queues,
reports, and audit logs should be written under an ignored path such as:

```text
reports/tmp_asml_offline_trial/
```

Generated artifacts must not be committed.

## Required ASML Company Context Fields

The ASML company context will be generated by the existing context builder after
source validation passes. The context should preserve:

- `schema_version`
- `ticker`: `ASML`
- `company_name`: `ASML Holding N.V.`
- `last_updated`
- `metrics`
- context-level `source_metadata`
- per-metric `metric_id`, `metric_name`, `metric_category`, `value`, `unit`,
  `period`, and nested `source_metadata`

Do not hand-edit generated context data unless the onboarding process requires
reviewing generated context output. Source metric records should remain the
primary input.

## Required Financial Metric Fields

Every ASML financial metric record must satisfy
`config/financial_metric_schema.json` and `config/source_rules.json`.

Required fields include:

- `company`
- `ticker`
- `metric_id`
- `metric_name`
- `metric_category`
- `value`
- `unit`
- `period`
- `accounting_basis`
- `statement_type`
- `source_url`
- `source_type`
- `source_date`
- `last_verified`
- `confidence`

Use `accounting_basis: "IFRS"` for IFRS-reported ASML financial statement
figures when that is what the official source supports. Use `Non-GAAP` only
when the source explicitly presents a non-GAAP measure, such as a manually
derived or company-presented free cash flow measure. Use `Other` for market
price snapshots or share-count records only when the current schema and source
meaning support it.

At minimum, the first source package should include the onboarding-required
metrics:

- `Revenue`
- `Net income`
- `Free cash flow`
- a `share_count` metric
- a `market_price` snapshot metric

Additional metrics such as gross profit, operating income, or segment metrics
may be added only when official source evidence is clear and every field is
validated.

## Required Source And Evidence Fields

Each ASML record must preserve:

- official source URL or reviewed source reference
- source type from the allowed enum
- source date
- last verified date
- unit and scale, such as `EUR millions` if verified
- fiscal period, including the fiscal year end date
- confidence value
- metric identifier stable enough for downstream references
- accounting basis and statement type

Market price snapshot records must also include:

- `currency`
- `exchange`
- `price_type`
- `as_of_datetime`
- `fetched_at`
- `provider`
- `retrieval_method`

The market price snapshot must be a manually recorded snapshot or reviewed file
import. It must not be fetched inside `run_analysis.py`, `model_rating.py`, DCF
logic, report generation, or any core module.

## Preferred Source Candidates

Preferred source candidates for manual review are:

- official ASML Investor Relations annual report materials
- official ASML annual report pages or annual report PDF
- official ASML results or investor relations pages when annual-report detail is
  insufficient
- SEC filing materials only if applicable and manually verified

Do not use unofficial data unless the source choice is separately justified in a
future data PR. Do not add a URL or number just because it appears plausible.
This plan intentionally does not validate or assert current ASML source URLs,
figures, source dates, or market prices.

Use placeholders until a maintainer manually verifies source material, for
example:

- `<official ASML annual report URL>`
- `<official ASML investor relations URL>`
- `<manual market price snapshot source URL>`

## Manual Verification Checklist Before Entering Numbers

Before any ASML number is entered into project data:

- confirm the source is official or explicitly justified
- confirm the source date and fiscal period
- confirm the reported currency and scale
- confirm whether the figure is IFRS, Non-GAAP, or Other
- confirm whether free cash flow is company-presented or derived from sourced
  cash-flow line items
- confirm the share-count metric meaning and period
- confirm the market price snapshot exchange, currency, timestamp, provider,
  retrieval method, and source date
- assign stable `metric_id` values
- record `last_verified`
- choose `confidence` conservatively
- verify that assumptions are labeled as assumptions, not facts
- keep any unresolved question as missing data or a research gap

If a field cannot be verified, do not invent it. Leave the trial blocked or mark
the output unavailable through the existing workflow.

## Validation Commands For A Future Trial

After ASML local files exist, run source/onboarding validation first:

```powershell
python scripts/validate_company_onboarding.py ASML --metrics-path data/asml_sample_metrics.json --dcf-assumptions-path data/companies/ASML/dcf_assumptions.json
```

If the trial does not include DCF assumptions yet, run source validation only:

```powershell
python scripts/validate_sources.py data/asml_sample_metrics.json
```

If validation fails, stop. Do not run analysis with invalid ASML source data.

## Analysis Command For A Future Trial

After validation passes, run ASML into an ignored reports path:

```powershell
python scripts/run_analysis.py ASML --source-data-path data/asml_sample_metrics.json --generate-report --generate-summary --run-dcf --dcf-assumptions-path data/companies/ASML/dcf_assumptions.json --reports-dir reports/tmp_asml_offline_trial/ASML --context-root reports/tmp_asml_offline_trial/companies --audit-log-path reports/tmp_asml_offline_trial/audit_log.jsonl --markdown-queue-path reports/tmp_asml_offline_trial/research_queue.md --json-queue-path reports/tmp_asml_offline_trial/research_queue.json
```

If DCF assumptions are not ready, omit `--run-dcf` and
`--dcf-assumptions-path`. Missing DCF, fair value per share, model rating, or
model signal outputs are acceptable when the reason is explicit and visible.

## ASML-Specific Risks To Watch

ASML trial review should pay special attention to:

- EUR versus USD units and any market price currency mismatch
- IFRS versus Non-GAAP labels
- ordinary share versus ADR or listed instrument differences
- exchange and ticker context for the market price snapshot
- share count definition and period
- annual report date versus fiscal period end date
- free cash flow definition and whether it is sourced or derived
- stale source or stale market price snapshots
- assumptions that require manual review and should block active model signals

These are data-quality and traceability risks, not investment conclusions.

## Guardrail Checks

ASML generated artifacts must not contain or imply:

- buy/sell/hold recommendations
- price targets
- personal investment advice
- trading instructions
- broker/order behavior
- portfolio automation
- hidden live-data fetching
- fabricated source claims
- provider API calls or credentials

Model rating, model confidence, and model signal outputs remain
non-personalized model artifacts only.

## Deferred Items

The following remain deferred and out of scope for the ASML offline trial:

- live data fetching
- provider APIs
- adapter implementation
- MCP/A2A runtime integration
- web UI or desktop app behavior
- methodology implementation
- generated artifact manifest implementation
- new financial logic or valuation formulas
- package rename or new runtime framework integration

## Smallest Safe Future PR Sequence

1. **Prepare ASML source package**: add only reviewed ASML source metrics,
   optional DCF assumptions, watchlist entry, and any required onboarding docs.
2. **Validate ASML source package**: run the onboarding/source validation
   commands and fix only evidence or formatting issues.
3. **Run ASML offline trial locally**: generate artifacts under
   `reports/tmp_asml_offline_trial/`, inspect them, and do not commit generated
   reports.
4. **Document ASML trial result**: summarize validation outcome, generated
   artifact review, guardrail review, missing data, and whether ASML is ready to
   become a supported sample ticker.

Do not combine ASML data entry with adapter work, live-data work, UI work,
methodology implementation, release notes, or financial-logic changes.
